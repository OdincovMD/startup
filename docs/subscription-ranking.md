# Подписки и ранжирование — продуктовая спецификация

---

## 1. Модель подписки

### Общие принципы

- Подписка привязана к **аккаунту пользователя** (User), а не к организации.
- **Два тарифа:** Basic и Pro.
- Активация через Admin API (ручная), без интеграции с платёжными системами в первой фазе.

### Таблица `user_subscriptions`

| Поле             | Тип                  | Описание                                                              |
|------------------|----------------------|-----------------------------------------------------------------------|
| `id`             | `SERIAL PRIMARY KEY` | Уникальный идентификатор                                              |
| `user_id`        | `INTEGER NOT NULL`   | Пользователь-владелец подписки (FK → `users.id`)                      |
| `audience`       | `VARCHAR NOT NULL`   | Тип роли: `"representative"`, `"researcher"`, `"student"`              |
| `tier`           | `VARCHAR NOT NULL`   | `"basic"` \| `"pro"` — тариф подписки                                 |
| `status`        | `VARCHAR NOT NULL`   | `"active"`, `"expired"`, `"cancelled"`                                |
| `started_at`     | `TIMESTAMPTZ`        | Дата и время начала подписки                                          |
| `expires_at`     | `TIMESTAMPTZ`        | Дата и время окончания подписки (NULL — бессрочная)                   |
| `trial_ends_at`  | `TIMESTAMPTZ`        | Окончание пробного периода (если есть)                               |
| `activated_by`   | `INTEGER`            | Администратор, активировавший подписку (FK → `users.id`)             |
| `cancelled_at`   | `TIMESTAMPTZ`        | Дата и время отмены (если была отменена)                              |
| `discount_percent` | `INTEGER`          | Скидка при продлении, % (для loyalty)                                 |
| `renewal_count`  | `INTEGER NOT NULL`   | Количество продлений                                                  |
| `created_at`     | `TIMESTAMPTZ`        | Дата создания записи                                                  |
| `updated_at`     | `TIMESTAMPTZ`        | Дата последнего обновления записи                                     |

### Тарифы

| Тариф | Ограничения | Дополнительно |
|-------|-------------|---------------|
| **Basic** | 1 организация **или** до 3 самостоятельных лабораторий | Boost в выдаче как у Pro |
| **Pro** | Без лимитов | Org-wide: лаборатории организации наследуют платный статус |

### Trial и Grace period

- **Trial:** при активации можно указать `trial_ends_at`. Пользователь считается платным до этой даты даже без `expires_at`.
- **Grace period:** первые 7 дней после создания первой org/lab — пользователь получает boost (верхний блок) без подписки. Стимулирует «попробовать и ощутить разницу».

### Таблица `subscription_events` (аудит)

| Поле              | Тип                  | Описание                                                              |
|-------------------|----------------------|-----------------------------------------------------------------------|
| `id`              | `SERIAL PRIMARY KEY` | Уникальный идентификатор                                              |
| `subscription_id` | `INTEGER NOT NULL`   | FK → `user_subscriptions.id`                                          |
| `event_type`      | `VARCHAR NOT NULL`   | `"activated"`, `"extended"`, `"expired"`, `"cancelled"`               |
| `performed_by`    | `INTEGER`            | Кто выполнил действие (FK → `users.id`)                               |
| `created_at`      | `TIMESTAMPTZ`        | Время события                                                        |
| `details`         | `JSONB`              | Произвольные данные: причина отмены, предыдущий `expires_at` и т. д. |

## 2. Правила распространения платного статуса

### Когда сущность считается платной (paid_active)

1. **Подписка активна:** `status = 'active'` и (`expires_at` пусто или в будущем).
2. **Trial активен:** `trial_ends_at` задан и в будущем.
3. **Grace period:** пользователь создал первую org/lab менее 7 дней назад и подписки нет.

### Базовый принцип

Платный статус сущности определяется по `creator_user_id`:

| Сущность     | Платная, если…                                                                 |
|--------------|--------------------------------------------------------------------------------|
| Организация  | Creator имеет активную подписку/trial/grace                                   |
| Лаборатория  | Creator имеет подписку/trial/grace **или** (см. Org-wide)                     |
| Вакансия     | Creator имеет подписку/trial/grace                                            |
| Запрос       | Creator имеет подписку/trial/grace                                            |

### Org-wide (тариф Pro)

> **Лаборатория, вступившая в организацию (OrgJoinRequest), наследует платный статус от организации, если creator организации имеет тариф Pro и активную подписку.**

| Организация | Тариф org creator | Лаборатория (вступила) | Статус лаборатории |
|-------------|-------------------|------------------------|--------------------|
| Платная     | Pro               | Бесплатная             | **Платная** (paid_via_org) |
| Платная     | Pro               | Платная (своя подписка) | **Платная**        |
| Платная     | Basic             | Бесплатная             | Бесплатная         |
| Бесплатная  | —                 | Платная                | Платная            |

**Вакансии и запросы:** статус всегда по `creator_user_id`, без наследования от организации.

## 3. Двухблочное ранжирование

### Общий принцип

Все каталоги используют двухблочную сортировку:

```
┌─────────────────────────────┐
│  Блок 1: Платные сущности   │  ← paid_active = true
│  (сортировка по rank_score) │
├─────────────────────────────┤
│  Блок 2: Бесплатные         │  ← paid_active = false
│  (сортировка по rank_score) │
└─────────────────────────────┘
```

### Поведение при поиске

| Тип запроса                  | Логика                                                                 |
|------------------------------|------------------------------------------------------------------------|
| **Без текстового поиска**    | `paid_active DESC, rank_score DESC, created_at DESC`                  |
| **Текстовый поиск (ES)**     | Релевантность — основа; платный boost зависит от `rank_score` (см. п. 8) |

## 4. Формулы ранжирования

### Антигейминг freshness

- Используется **`first_created_at`** (не перезаписывается при обновлениях) — пересоздание сущности не возвращает максимум баллов.
- Сущности **0–2 дней** получают сниженный freshness, чтобы не поощрять перепосты и спам.

### Longevity bonus

- Для org и lab: **+1 балл за каждые 6 месяцев** «жизни» сущности (по `first_created_at`), **максимум 5**.
- Поощряет долгосрочную активность.

---

### 4.1. Organization Score (0–100)

#### quality (0..45)

| Баллы | Условие                                  |
|-------|------------------------------------------|
| +8    | Заполнен `avatar_url`                    |
| +10   | `description` ≥ 300 символов             |
| +6    | Заполнен `website`                       |
| +8    | Заполнен `ror_id`                        |
| +5    | Заполнен `address`                       |
| +8    | Есть хотя бы 1 опубликованная лаборатория |

#### freshness (0..15)

По `first_created_at`:

| Возраст записи | Баллы |
|----------------|-------|
| 0–2 дня        | **4** (антигейминг)                      |
| 3–14 дней      | 15    |
| 15–30 дней     | 10    |
| 31–90 дней     | 6     |
| > 90 дней      | 2     |

#### performance (0..20)

| Баллы | Сигнал                                      |
|-------|---------------------------------------------|
| +10   | `log1p(unique_views_30d)` нормализованный   |
| +10   | `avg_time_on_page_sec` нормализованный       |

#### structure (0..20)

| Баллы | Формула                                          |
|-------|--------------------------------------------------|
| +8    | `min(labs_count, 4) * 2`                         |
| +6    | `min(employees_count, 6)`                        |
| +6    | `min(vacancies_count + queries_count, 6)`        |

#### longevity (0..5)

| Баллы | Формула                               |
|-------|--------------------------------------|
| до 5  | `min(5, age_months // 6)`            |

**Итоговая формула:**

```
org_rank_score = min(100, quality + freshness + performance + structure + longevity)
```

## 5. Главная страница

Главная страница использует каталожные API с сортировкой:

```
paid_active DESC, rank_score DESC, created_at DESC
```

Платные сущности отображаются первыми без дополнительной логики.

### Будущее развитие

- Featured-эндпоинты с ротацией.
- Impression tracking для честного распределения показов.

## 6. Разрешение ничьих

| Ситуация                    | Поведение                                                        |
|-----------------------------|------------------------------------------------------------------|
| Одинаковый `rank_score`     | Сортировка по `created_at DESC`                                  |
| Истечение подписки         | Сущность **немедленно** перемещается в бесплатный блок          |
| Нет подписки               | Бесплатный блок, сортировка по `rank_score`                      |

## 7. Дифференцированный boost при поиске

При **текстовом поиске** платный статус учитывается через boost, зависящий от качества карточки:

```
paid_boost = 1.0 + (rank_score / 100) * 1.5   // диапазон 1.0–2.5
```

- Хорошо заполненные платные карточки получают больший boost.
- Слабо заполненные платные не доминируют над качественными бесплатными.
- Стимулирует улучшение профилей.

## 8. Elasticsearch

### Поля в маппингах

Для **всех 4 индексов** (organizations, laboratories, vacancies, queries):

| Поле               | Тип       | Описание                                           |
|--------------------|-----------|----------------------------------------------------|
| `paid_active`      | `boolean` | Активная подписка/trial/grace у создателя         |
| `rank_score`      | `float`   | Скор ранжирования (0–100)                          |
| `creator_user_id` | `integer` | ID создателя                                       |
| `first_created_at`| `date`    | Первая дата создания (для freshness и longevity)  |

Дополнительно для **лабораторий**:

| Поле                | Тип       | Описание                                  |
|---------------------|-----------|-------------------------------------------|
| `org_creator_user_id` | `integer` | Creator организации (для org-wide)       |
| `paid_via_org`      | `boolean` | Платный статус унаследован от организации |

### Поведение при поиске

**С текстовым запросом** — `script_score` для платного boost:

```javascript
// boost = 1.0 + (rank_score/100) * 1.5 для paid_active
if (paid_active) return 1.0 + (rank_score / 100.0) * 1.5;
return 1.0;
```

**Без текстового запроса** — сортировка:

```json
[
  { "paid_active": { "order": "desc", "missing": "_last" } },
  { "rank_score":  { "order": "desc", "missing": "_last" } },
  { "created_at":  { "order": "desc" } }
]
```

## 9. Дополнительные возможности

### Онбординг и подсказки

- В дашборде и карточках отображаются **подсказки по улучшению** коэффициента (tips).
- Блок «Как подняться в выдаче» на странице подписки.
- Подробные tips по каждой сущности — в разделе «Аналитика».

### Расширенная аналитика (Pro)

- Эндпоинт `/profile/analytics/extended` доступен только подписчикам Pro.
- Данные: просмотры за 7/30 дней, конверсия, среднее время на странице.
- Сравнительный контекст и рекомендации по улучшению.

### Loyalty discount

- При продлении увеличивается `renewal_count`.
- Поле `discount_percent` — для применения скидки (ручная настройка или будущая интеграция с биллингом).

| Механизм                  | Цель                                                     |
|---------------------------|----------------------------------------------------------|
| **Тарифы Basic / Pro**    | Разные цены и объёмы; входной тариф для малых игроков    |

| **Антигейминг freshness**| Меньше перепостов и спама; честное соревнование         |
| **Boost от rank_score**  | Улучшение карточек; качественный контент в топе         |
| **Longevity bonus**      | Поддержка долгосрочных партнёров; снижение churn        |
| **Онбординг / tips**     | Понятный путь улучшения; рост заполненности профилей    |
