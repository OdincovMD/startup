# Подписки и ранжирование — продуктовая спецификация

---

## 1. Модель подписки

### Общие принципы

- Подписка привязана к **аккаунту пользователя** (User), а не к организации.
- Уровней подписки нет — существует единственный платный уровень.
- Первая фаза: ручная активация через Admin API, без интеграции с платёжными системами.

### Таблица `user_subscriptions`

| Поле           | Тип                  | Описание                                                  |
|----------------|----------------------|-----------------------------------------------------------|
| `id`           | `SERIAL PRIMARY KEY` | Уникальный идентификатор                                  |
| `user_id`      | `INTEGER NOT NULL`   | Пользователь-владелец подписки (FK → `users.id`)          |
| `audience`     | `VARCHAR NOT NULL`   | Тип роли: `"representative"`, `"researcher"`, `"student"` |
| `status`       | `VARCHAR NOT NULL`   | `"active"`, `"expired"`, `"cancelled"`                    |
| `started_at`   | `TIMESTAMPTZ`        | Дата и время начала подписки                              |
| `expires_at`   | `TIMESTAMPTZ`        | Дата и время окончания подписки                           |
| `activated_by` | `INTEGER`            | Администратор, активировавший подписку (FK → `users.id`)  |
| `cancelled_at` | `TIMESTAMPTZ`        | Дата и время отмены (если была отменена)                  |
| `created_at`   | `TIMESTAMPTZ`        | Дата создания записи                                     |
| `updated_at`   | `TIMESTAMPTZ`        | Дата последнего обновления записи                         |

### Таблица `subscription_events` (аудит)

| Поле              | Тип                  | Описание                                                              |
|-------------------|----------------------|-----------------------------------------------------------------------|
| `id`              | `SERIAL PRIMARY KEY` | Уникальный идентификатор                                              |
| `subscription_id` | `INTEGER NOT NULL`   | FK → `user_subscriptions.id`                                          |
| `event_type`      | `VARCHAR NOT NULL`   | `"activated"`, `"extended"`, `"expired"`, `"cancelled"`               |
| `performed_by`    | `INTEGER`            | Кто выполнил действие (FK → `users.id`)                               |
| `created_at`      | `TIMESTAMPTZ`        | Время события                                                        |
| `details`         | `JSONB`              | Произвольные данные: причина отмены, предыдущий `expires_at` и т. д. |

## 2. Правила распространения платного статуса

### Принцип определения статуса

Платный статус сущности определяется **исключительно** по наличию активной подписки у `creator_user_id` этой сущности:

| Сущность     | Платная, если…                                             |
|--------------|------------------------------------------------------------|
| Организация  | `creator_user_id` организации имеет активную подписку      |
| Лаборатория  | `creator_user_id` лаборатории имеет активную подписку      |
| Вакансия     | `creator_user_id` вакансии имеет активную подписку         |
| Запрос       | `creator_user_id` запроса имеет активную подписку          |

### Критическое правило: вступление через OrgJoinRequest

> **Вступление в организацию через OrgJoinRequest НЕ передаёт платный статус.**
>
> Лаборатория, вступившая в платную организацию, **НЕ** становится платной.
> Поле `organization_id` используется для структуры, контроля доступа, фильтров и отображения — **НЕ** для наследования подписки.

### Матрица статусов

| Организация | Присоединённая лаборатория | Статус организации | Статус лаборатории |
|-------------|---------------------------|--------------------|--------------------|
| Платная     | Бесплатная (вступила)     | **Платная**        | **Бесплатная**     |
| Платная     | Платная (вступила)        | **Платная**        | **Платная** (своя подписка) |
| Бесплатная  | Платная                   | **Бесплатная**     | **Платная**        |

**Вакансии и запросы:** статус определяется по `creator_user_id`, а не по `organization_id`.

## 3. Двухблочное ранжирование

### Общий принцип

Все каталоги используют двухблочную сортировку:

```
┌─────────────────────────────┐
│  Блок 1: Платные сущности   │  ← paid_active = true
│  (сортировка по rank_score) │
├─────────────────────────────┤
│  Блок 2: Бесплатные         │  ← paid_active = false
│  (сортировка по rank_score) │
└─────────────────────────────┘
```

### Поведение в зависимости от типа запроса

| Тип запроса                  | Логика сортировки                                                                                 |
|------------------------------|---------------------------------------------------------------------------------------------------|
| **Каталог без текстового поиска** | `paid_active DESC, rank_score DESC, created_at DESC`                                              |
| **Текстовый поиск (ES)**         | ES-релевантность — основной фактор; платный статус и качество применяются как boost через `function_score` |

## 4. Формулы ранжирования

### 4.1. Organization Score (0–100)

#### quality (0..45)

| Баллы | Условие                                  |
|-------|------------------------------------------|
| +8    | Заполнен `avatar_url`                    |
| +10   | `description` ≥ 300 символов             |
| +6    | Заполнен `website`                       |
| +8    | Заполнен `ror_id`                        |
| +5    | Заполнен `address`                       |
| +8    | Есть хотя бы 1 опубликованная лаборатория |

#### freshness (0..15)

| Возраст записи | Баллы |
|----------------|-------|
| 0–14 дней      | 15    |
| 15–30 дней     | 10    |
| 31–90 дней     | 6     |
| > 90 дней      | 2     |

#### performance (0..20)

| Баллы | Сигнал                                      |
|-------|---------------------------------------------|
| +10   | `log1p(unique_views_30d)` нормализованный   |
| +10   | `avg_time_on_page_sec` нормализованный       |

#### structure (0..20)

| Баллы | Формула                                          |
|-------|--------------------------------------------------|
| +8    | `min(labs_count, 4) * 2`                         |
| +6    | `min(employees_count, 6)`                        |
| +6    | `min(vacancies_count + queries_count, 6)`        |

**Итоговая формула:**

```
org_rank_score = quality + freshness + performance + structure
                 (0..45)   (0..15)     (0..20)       (0..20)    = 0..100
```

### 4.3. Vacancy Score (0–100)

#### relevance (0..35)

> Применяется **только** при текстовом поиске.
> ES `_score` нормализуется в диапазон 0..35.

#### quality (0..25)

| Баллы | Условие                                      |
|-------|----------------------------------------------|
| +7    | Заполнены `requirements`                     |
| +7    | Заполнено `description`                      |
| +4    | Указан `employment_type`                     |
| +4    | Указан `contact_email` или `contact_phone`   |
| +3    | Привязана лаборатория или организация        |

#### freshness (0..20)

| Возраст записи | Баллы |
|----------------|-------|
| 0–7 дней       | 20    |
| 8–21 день      | 14    |
| 22–45 дней     | 8     |
| > 45 дней      | 3     |

#### performance (0..20)

| Баллы | Сигнал                                                              |
|-------|---------------------------------------------------------------------|
| +8    | `log1p(unique_viewers)` нормализованный                             |
| +8    | `smoothed_conversion = (responses + 3) / (unique_views + 30)`      |
| +4    | `avg_time_on_page_sec` нормализованный                              |

**Итоговая формула:**

```
vacancy_rank_score = relevance + quality + freshness + performance
                     (0..35)     (0..25)   (0..20)     (0..20)      = 0..100
```

## 5. Главная страница / Рекомендуемое

### Текущее поведение

Главная страница берёт первые N элементов из каталожных API.

### Рекомендуемый подход

Сделать дефолтную сортировку каталогов subscription-aware:

```
ORDER BY paid_active DESC, rank_score DESC, created_at DESC
```

Таким образом, главная страница **автоматически** показывает платные сущности первыми без дополнительной логики.

### Будущее развитие (опционально)

- Отдельные featured-эндпоинты с ротацией, чтобы одни и те же сущности не занимали верхние позиции постоянно.
- Impression tracking для справедливого распределения показов.

## 6. Разрешение ничьих и fallback-поведение

| Ситуация                    | Поведение                                                        |
|-----------------------------|------------------------------------------------------------------|
| Одинаковый `rank_score`     | Сортировка по `created_at DESC`                                  |
| Истечение подписки          | Сущность **немедленно** перемещается в бесплатный блок           |
| Нет подписки                | Бесплатный блок, обычная сортировка по `rank_score`              |

## 8. Реализация в Elasticsearch

### Новые поля в маппингах

Добавляются во **все 4 индекса** (organizations, laboratories, vacancies, queries):

| Поле               | Тип       | Описание                                              |
|--------------------|-----------|-------------------------------------------------------|
| `paid_active`      | `boolean` | Есть ли у создателя активная подписка                 |
| `rank_score`       | `float`   | Вычисленный скор ранжирования (0–100)                 |
| `creator_user_id`  | `integer` | ID создателя — для lookup подписки при переиндексации |

### Поведение при поиске

**С текстовым запросом** — оборачиваем в `function_score`:

```json
{
  "function_score": {
    "query": { "...": "основной текстовый запрос" },
    "functions": [
      {
        "filter": { "term": { "paid_active": true } },
        "weight": 2
      }
    ],
    "boost_mode": "multiply",
    "score_mode": "sum"
  }
}
```

ES-релевантность остаётся основным фактором; платный статус и качество работают как boost.

**Без текстового запроса** — жёсткая сортировка:

```json
{
  "sort": [
    { "paid_active": { "order": "desc" } },
    { "rank_score":  { "order": "desc" } },
    { "created_at":  { "order": "desc" } }
  ]
}
```
